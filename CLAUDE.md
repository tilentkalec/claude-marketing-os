# claude-marketing-os

An AI-native marketing operating system. A CMO orchestrator with channel agents (Meta, Google, Email, SEO, Social) you configure once and run for any brand. Built on Claude Code.

## What this is

A repo skeleton that lets one operator run a multi-channel marketing function with AI agents instead of hiring a full agency. You configure the brand once via `client-config.md`, and every agent reads from that single source of truth.

The architecture:

```
                    User
                      │
                      ▼
              CMO Orchestrator  (AGENTS.md)
                      │
       ┌──────────────┼──────────────┐
       ▼              ▼              ▼
   Channel agents (.claude/agents/)
   ─ content-strategist
   ─ content-producer
   ─ social-media
   ─ meta-ads
   ─ google-ads
   ─ email
       │
       ▼
   Skills (skills/)        ← reusable playbooks
       │
       ▼
   client-config.md        ← single source of brand truth
```

## How to start

1. **Configure the brand** — run the `client-onboarding` skill (or just chat with the CMO; it'll route there if `client-config.md` is missing). This generates `client-config.md` at the repo root.
2. **Add credentials** — copy `.env.example` to `.env` and fill in the API keys for the channels you actually run.
3. **Talk to the CMO** — type `/cmo-report` for a weekly cross-channel review, `/cmo-analyze "<question>"` for a targeted question, or `/cmo-plan "<goal>"` for strategic planning.

## How agents and skills relate

- **Agents** live in `.claude/agents/`. They're channel-shaped — one per marketing function. Each one reads `client-config.md`, then loads the appropriate skill.
- **Skills** live in `skills/`. They're playbooks — pure operational knowledge, brand-agnostic. You can use them directly (e.g., `/skill blog-post-writer`) or let an agent dispatch them.

This separation means you can fork the repo, change `client-config.md` for a new brand, and the agents work without modification.

## Data layer

This repo doesn't ship with data-pulling scripts. You bring your own:
- Traffic/conversions: GA4, Plausible, Fathom — anything that exposes an API
- Revenue: Stripe, WooCommerce, Shopify, your own backend

Drop scripts in `scripts/`, dump JSON to `data/`, and the CMO will read from there. See `docs/data-sources.md` for the JSON contract the orchestrator expects.

## What lives where

```
.
├── AGENTS.md                    ← CMO orchestrator (top-level instructions)
├── CLAUDE.md                    ← this file
├── README.md                    ← project README
├── client-config.md             ← brand config (gitignored, generated by onboarding)
├── .env                         ← credentials (gitignored)
├── .env.example                 ← credential template
├── .claude/
│   ├── agents/                  ← 6 channel subagents
│   └── commands/                ← /cmo-report, /cmo-analyze, /cmo-plan
├── skills/                      ← reusable playbooks
├── data/                        ← traffic/revenue JSON (gitignored)
├── outputs/                     ← reports, plans, generated content (gitignored)
├── notes/                       ← optional persistent client notes (gitignored)
├── scripts/                     ← user's data-pulling scripts
└── docs/                        ← architecture and design docs
```

## Privacy / data hygiene

Anything client-specific (`client-config.md`, `.env`, `data/`, `outputs/`, `notes/`) is gitignored. The repo template stays brand-agnostic; client work stays local.

If you want to share a fork of this template that's been customized for a specific client, scrub these directories first.
